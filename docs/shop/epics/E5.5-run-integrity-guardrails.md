# E5.5 — Run Integrity Guardrails (Confidence-Based Anti-Abuse)

## Goal

Reduce trivial score/continue abuse and improve leaderboard trust using practical server-side run validation + confidence scoring, without overengineering full anti-cheat.

## Scope

- Introduce server-issued run sessions (`runId`) as the required unit for score submission.
- Enforce one-time continue usage server-side.
- Allow score finalize update only once under strict bounds.
- Add confidence script to classify submitted scores (`clean`, `suspicious`, `impossible`).
- Add lightweight anomaly logging and moderation hooks.

## Deliverables

- `space-server` run session endpoints and persistence:
  - `POST /runs/start`
  - `POST /runs/:runId/continue` (max once)
  - `POST /runs/:runId/finalize`
  - `PATCH /runs/:runId/finalize` (max one update)
- Run/session DB fields:
  - `run_id`, `player_id`, `started_at`, `finalized_at`, `continue_used`, `finalize_update_count`
  - `shop_snapshot` (or equivalent)
  - `score_submitted`, `score_final`
  - `confidence_status`, `confidence_reasons`
- Confidence evaluator script/service based on gameplay envelope + shop snapshot.
- Admin/moderation visibility for flagged rows.

## Acceptance Criteria

- Client cannot submit a score without a valid server-issued `runId`.
- Continue can be consumed at most once per run on server record.
- Finalize update is accepted at most once, only if:
  - same `runId`
  - update window valid
  - score monotonic increase
  - run not locked/closed
- Confidence evaluator marks impossible outliers and stores reasons.
- Flagged entries are reviewable (and optionally hidden from public leaderboard by policy).

## Dependencies

- E2 offline model stable.
- E5 server sync baseline in place (or implemented in parallel as shared foundation).

## Out of Scope

- Full device attestation / jailbreak detection.
- Impossible-to-tamper offline gameplay.
- Enterprise anti-cheat stack.
